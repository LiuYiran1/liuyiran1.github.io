### 优化

#### 支配分析

- 支配：如果从 **入口节点** 到 **节点 $n$** 的**每一条路径**都必须经过 **节点 $d$**，那么我们就说 **$d$ 支配 $n$**（记作 $d \text{ dom } n$）。

- 严格支配：$$SDom(n) = \{ d \in \text{Nodes} \mid d \text{ dom } n \wedge d \neq n \}$$

- 直接支配：

  对于一个非入口节点 $n$，它的**直接支配者** $IDom(n)$ 是 $SDom(n)$ 中满足以下条件的唯一节点：

  **条件：**
  			$$IDom(n) \in SDom(n)$$
  且对于 $SDom(n)$ 中的任意节点 $d$（只要 $d \neq IDom(n)$），都有：
  			$$d \text{ dom } IDom(n)$$

- 支配树：

  在支配树中，入口节点是根节点。对于每个节点 $n$，它的父亲节点就是 $IDom(n)$。

  *   **意义：** 支配树把复杂的控制流（可能有环、有分支）简化成了清晰的层级结构。如果在支配树中 $d$ 是 $n$ 的祖先，那么在 CFG 中 $d$ 一定支配 $n$。

- 支配边界（DF）
  节点 $X$ 的支配边界节点 $Y$ 满足以下条件：

  1.  $X$ 支配 $Y$ 的某个前驱节点（即 $X$ 的管辖范围延伸到了 $Y$ 的门口）。
  2.  $X$ **不严格支配** $Y$。

  - 支配边界告诉我们一个节点的支配权在哪里失效了
  - 为啥要知道支配边界？
    - 它是构建 **SSA（Static Single Assignment）** 形式的关键。如果在节点 $X$ 中定义了一个变量 $a$，那么在 $X$ 支配权失效的地方（即 $DF(X)$ 中的每个节点），我们可能需要插入一个 **$\Phi$ 函数** 来合并不同路径传过来的变量值。

- **我们使用的算法：**最基础的迭代式算法

  - 第一步：不断应用以下公式，找到每个点的必经集合 $Dom(n)$
    $$Dom(n) = \{n\} \cup \left( \bigcap_{p \in preds(n)} Dom(p) \right)$$
  - 第二步：找到直接支配
  - 第三步：构造支配树
  - 第四步：构造支配边界
    - 先找出 CFG 中的有多个前驱的节点的节点 $b$
      - 遍历 $b$ 的每一个前驱节点 $p$。
      - 沿着支配树向上走：从 $p$ 开始，一直走到 $IDom(b)$ 为止。
      - 在这个路径上的所有节点（不包括 $IDom(b)$），它们的支配边界集合里都要加上 $b$。

  - **这个算法的时间复杂度是 $O(N^2)**：有更好的 **Lengauer-Tarjan 算法**

#### 常量传播

SCCP 稀疏条件常量传播

- 给每个变量定义一个 Lattice：OverDEF<-Constant<-UNDEF

- 使用两个工作表来实现：

  1. flow worklist ：记录哪些基本块是可达的
     - 一开始就只有entry在里面，处理一个跳转语句的时候，如果cond是常量的话就把对应的基本块加到队列，如果不是就把所有后继都加入队列

  2. SSA worklit ：记录哪些指令需要重新计算
     - 如果某个变量的值的状态变化了，那所有使用这个变量的指令都要进入队列重新算一遍

- SCCP不会去走死代码，所以死代码中的逻辑不会影响到“某个变量是常量“的正确结论

- 还有啥常量传播？稀疏常量传播（基于SSA，但是不会处理死代码），过程间常量传播，位级常量传播（不仅关注变量1是否是常量，还关注变量的某一位，变量可能不是常量但是某一位是常量，之后做位运算的时候就能得到常量），范围传播

#### mem2reg

**目的：把分配在栈上的局部变量提升到寄存器中，从而将程序转换为SSA形式**

- 提升的变量类型有：int、float、pointer
- **算法逻辑：**
  1. 找出所有对变量 x 执行了 store 操作的基本块（DefBlocks）
  2. 对于每一个 DefBlock ，都在该块的支配边界中的每一个基本块中插入 $\Phi$ 指令
     - 支配边界就是支配权的终止处，也是多条路径交汇的起点，所以使用 $\Phi$ 节点来选择
  3. 插入 $\Phi$ 节点本身也算作一次对该变量的“定义（store）”，因此可能引发新的 $\Phi$ 节点插入（直到达到固定点）
  4. 变量重命名：一个回溯算法
     1. **进入一个块时：**
        *   如果块开头有 $\Phi$ 节点，这些 $\Phi$ 的结果就是该变量最新的值，压入对应的栈。
        *   **处理指令**：
            *   遇到 `Store(value, ptr)`：这个 `value` 变成了最新的值，压入变量 `ptr` 对应的栈。删掉这个 Store。
            *   遇到 `Load(ptr)`：直接把这个 Load 替换成此时栈顶的值。删掉这个 Load。
     2.  **填充后继块的 $\Phi$ 参数：**
         *   看当前块的所有**后继块**。如果后继块里有 $\Phi$ 节点，就把当前栈顶的值填进 $\Phi$ 的对应参数位置（告诉 $\Phi$：如果你从我这走过去，变量就是这个值）。
     3.  **递归遍历**：按支配树顺序去处理子节点（因为支配者定义的变量在被支配者里才有效）。
     4.  **回退（清理栈）**：
         *   当处理完一个块及其所有子块，准备离开时，要将在该块中压入栈的值全部弹出（Pop）。
         *   **为什么？** 为了保证分支 A 的赋值不会污染到并列的分支 B。

#### 循环分析

找到代码中的自然循环，什么是自然循环？如果有一条边 $pred \to bb$，且 **$bb$ 支配 $pred$**，那么这条边就是一条回边。$bb$ 就是循环的 Header

- **算法：**
  1. 寻找回边
     - 遍历所有边：$a->b$ ，如果 $b$ 支配 $a$ 的话就认定 $b$ 是 header
  2. 收集循环体：从 a 开始往上找，直到找到 b 得到循环包含的基本块
  3. 规范化：如果 header 有好几个前驱，就创建一个preheader来接收这些前驱，然后让preheader作为header的唯一前驱



SSA 形式的挑战：LCSSA

在 SSA 形式下（即你之前的 `Mem2Reg` 之后），循环优化变得非常脆弱。

假设你在循环里算出了一个值 `%v1`，循环结束后外面还在用它。如果你把循环展开了，或者移动了指令，外面那个使用点可能就找不到正确的定义了。

为了防止这种破坏，编译器通常会引入 **LCSSA (Loop-Closed SSA)**：
*   **做法**：在循环的所有出口块（Exit Blocks）插入特殊的 $\Phi$ 节点。
*   **目的**：这像是一层“防火墙”，把循环内部的变量和外部隔离开。这样即使循环内部结构变了，只要防火墙还在，外部引用就不会崩溃。

#### 循环不变量外提

- **算法：**

  1. 判断什么是不变量

     - 先找可以外提的指令，比如内存操作(load, strore)，函数调用(Call)，跳转指令

       - **这里还有个雷区：**

         ```java
         while (cond) {
             if (x != 0) {
                 res = 100 / x; // x 在循环外定义，比如 x = 0
             }
         }
         ```

         如果把 `res = 100 / x` 外提，如果 cond 为真但 x 为 0，原本程序在循环内因为 if 保护不会崩溃，但被提到外面后，程序**一进循环前就除零崩溃了**，所以外提指令时，必须保证该指令所在的块**支配了循环所有的出口**，或者证明该指令**绝不会触发异常**

     - 操作数稳定性检查，如果一个指令用到的所有东西（操作数）要么是**常量**，要么是**在循环外面定义的**，那么这个指令在循环里的结果就一定是恒定的

  2. 迭代到不动点

     - 因为不变量会传递，直到没有不变量能被提到 preHeader 中算法才停止
